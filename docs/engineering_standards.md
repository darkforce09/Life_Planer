# Modern Engineering & Coding Standards

This document outlines the strict, modern programming methods and documentation rules that must be followed. We will use the latest, state-of-the-art standards for TypeScript development to ensure the system is extremely robust and fast.

## 1. Programming Paradigm & Architecture
- **Modern TypeScript (ESM):** The backend will be built in **TypeScript (Node.js)** using **ES Modules (ESM)**. Strict mode is mandatory (`"strict": true`).
- **Functional-Core / Imperative-Shell:** While we will use Classes to encapsulate stateful API integrations, the core business logic (like the Prioritization Engine math) should be written as pure, easily testable functional methods.
- **Dependency Injection:** Database connections, logger instances, and headless browser contexts will be passed into classes/functions via their constructors to ensure high modularity.

## 2. Data Validation & Type Safety
- **Runtime Validation (Zod):** TypeScript only protects you at compile time. All external data coming from the "Sensors" (Canvas APIs, TimeEdit ICS files) **MUST** be validated at runtime using **Zod** schemas before entering our system. We do not trust external university APIs.
- **End-to-End Type Safety:** The database schema types will be exported and shared with the UI widget to ensure absolute E2E type safety.

## 3. Database Interaction
- **Modern ORM (Drizzle):** We will use **Drizzle ORM** to interact with PostgreSQL. Drizzle provides maximum type safety, superior performance, and edge-compatibility without the heavy overhead of older ORMs.
- Database schemas will be managed via strict migration files generated by Drizzle.

## 4. Code Documentation (TSDoc)
- All classes, methods, and complex logic blocks must be documented using strict **TSDoc** standards (the modern standard for TypeScript).
- **Inline Comments:** Inline comments should only be used to explain *why* a specific decision was made (business logic), not *what* the code is doing (the code should be clean enough to be self-documenting).

## 5. Testing & Debugging Standards
- **Modern Unit Testing (Vitest):** We will use **Vitest** (which is significantly faster and has native ESM support) instead of older tools like Jest.
- **Scraper Resiliency (Playwright):** Because the system relies on fragile web scraping, we will write **Playwright** integration tests that run daily to verify the scrapers haven't broken due to university UI changes.
- **Embedded Debug Systems & Telemetry:** 
  - Never use raw `console.log()` in production.
  - We will implement a modern structured logger (**Pino**, because it's extremely fast) that tags logs by domain (e.g., `[SENSOR-CANVAS]`, `[AGENT-LADOK]`).
  - **Debug Scripts:** Every complex module must have an associated run script for manual debugging (e.g., an npm script to run the Playwright scraper visibly on your screen).
  - Errors must be securely saved and pushed to a "System Health" view in the Face widget.

## 6. Elite Resilience & Security Standards
- **CI/CD & Deployment:** We will use GitHub Actions (or GitLab CI) for Continuous Integration. Every push will trigger the Vitest and Playwright test suites. Passing builds will automatically deploy to the self-hosted server using Docker.
- **Auto-Healing Scrapers:** If a Playwright scraper fails to find a DOM element, it must implement a Vision AI fallback. It will take a screenshot, pass it to a Vision model (like Gemini), and ask for the new coordinates of the target button.
- **Strict Data Sanitization (Privacy):** Before any data is sent to a cloud LLM (e.g., for drafting emails), a dedicated sanitization class must strip out all Personal Identifiable Information (PII), such as personal numbers, passwords, and sensitive medical data.
